# PostgreSQL 대용량 데이터 성능 최적화 및 아키텍처 설계 실습

500만 건 이상의 대규모 데이터를 활용하여 인덱스(Index) 최적화와 테이블 파티셔닝(Partitioning) 전략을 실습하고,   
성능 향상과 시스템 부하 사이의 트레이드오프(Trade-off)를 데이터로 증명한 프로젝트입니다.

---

## 데이터 준비 및 실행 방법 (How to Setup)

이 프로젝트는 데이터의 용량이 크기 때문에 데이터베이스 덤프 파일을 직접 제공하지 않습니다.   
대신, 제공된 Python 스크립트를 통해 누구나 동일한 테스트 환경(500만 건)을 구축할 수 있습니다.  

1. **사전 준비**  
   - PostgreSQL 12버전 이상 설치 및 실행
   - Python 3.x 설치
   - `psycopg2`, `Faker` 라이브러리 설치 (`pip install psycopg2 faker`)

2. **데이터 적재**  
   - `sql_tuning.py`를 실행하여 500만 건의 데이터를 로컬 데이터베이스에 생성합니다.
   ```bash
   python sql_tuning.py

3. **성능 측정**  
   - 각 스크립트 상단의 DB_CONFIG 정보를 본인의 DB 환경에 맞게 수정한 후 실행하여 성능 지표를 확인합니다.

---
   
## 1. 쿼리 성능 최적화 (Indexing)
특정 사용자 ID 조회 시 발생하는 **Full Table Scan** 문제를 B-Tree 인덱스를 통해 해결하였습니다.

- **문제 상황**: 500만 건 전수 조사로 인해 약 265ms 소요.
- **해결 방안**: `user_id`, `status` 컬럼에 대한 복합 인덱스(Composite Index) 생성.
- **성과**: 조회 속도 **265.6ms → 1.1ms (약 225배 향상)**.
- **인사이트**: `Filter` 단계를 제거하고 `Heap Blocks` 접근을 4배 이상 줄여 I/O 효율성을 극대화함.

---

## 2. 인덱스 오버헤드 분석 (Write Overhead)
조회 성능 향상을 위해 생성한 인덱스가 쓰기(DML) 성능에 미치는 악영향을 수치로 분석했습니다.

- **실험**: 인덱스 3개 상태 vs 5개 상태에서 10만 건 삽입(INSERT) 시간 비교.
- **결과**:
  - 인덱스 3개: **1.38초**
  - 인덱스 5개: **13.27초**
- **성과**: 인덱스 2개 추가 시 쓰기 성능이 **861.79% 저하**됨을 증명.
- **인사이트**: 무분별한 인덱스 생성은 쓰기 병목의 주원인이 됨을 파악, 주기적인 'Unused Index' 모니터링의 필요성 체감.

---

## 3. 대용량 설계 전략 (Table Partitioning)
데이터가 지속적으로 증가하는 환경을 고려하여 월별(Monthly) 테이블 파티셔닝을 도입했습니다.

- **기법**: `Range Partitioning` (기준: `order_date`)
- **성과**: **Partition Pruning**을 통해 500만 건 중 필요한 월의 데이터(약 82만 건)만 스캔하도록 최적화.
- **인사이트**: 물리적 테이블 분할을 통해 거대 인덱스 관리 부하를 줄이고 데이터 가용성을 확보함.

---

## 인사이트
- 인덱스는 조회 성능을 비약적으로 높이지만, 쓰기 성능과는 반비례 관계에 있음을 실제 수치로 확인했습니다.
- 선택도(Selectivity)가 높은 컬럼을 인덱스 앞쪽에 배치하는 설계의 중요성을 학습했습니다.
- 데이터 규모에 따라 인덱스만으로 해결되지 않는 문제는 파티셔닝 아키텍처로 대응해야 함을 실습했습니다.
